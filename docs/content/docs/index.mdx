Chimely is a self-hostable **in-app notification inbox**: one Rust binary +
Postgres + Redis, a small HTTP API, and a drop-in `<Inbox />` React component.

It is the inbox primitive, not a workflow engine. There are no flows, steps,
or channel routers. You send notifications; Chimely stores, fans out, counts,
and streams them; the widget renders them. The whole integration is:

- **One POST to notify** — your backend calls `POST /v1/notifications`.
- **One `<Inbox />` to render** — drop the component into your frontend.

<Callout type="warn" title="Pre-1.0">
Chimely is at **0.1.0**. The HTTP API and the SDK surface may change on minor
version bumps until 1.0.0. Pin your versions.
</Callout>

## Architecture

Postgres is the source of truth. Redis is the real-time hint plane and is
recomputable from Postgres at any time. The browser never talks to your
database.

```
your backend ──POST /v1/notifications──▶ Chimely ──┬─ Postgres (authoritative)
                                                    └─ Redis (hints/cache)
                                                          │
your frontend ◀── <Inbox /> ◀── SSE hint + REST refetch ──┘
```

- **Postgres ≥15** holds every notification, broadcast, subscriber, watermark,
  and counter. Recomputable, partitioned, retained.
- **Redis** carries SSE hints and cached counters. Losing Redis delays hints,
  never loses data. Chimely also runs Redis-less on a single node (hints ride
  Postgres `LISTEN/NOTIFY`). See [Self-hosting](/docs/self-hosting).
- **SSE is a hint, not a transport.** Clients refetch over REST (conditional,
  ETag) on every hint. A missed hint is harmless by construction.

## The two-source inbox

The inbox merges two sources, and the list, the unread count, and read state
agree across both at all times:

| Source | Fan-out | Row model |
|---|---|---|
| **Direct notification** | on write | one row per recipient |
| **Broadcast** | on read | one announcement row, never materialized per subscriber |

A subscriber sees a broadcast iff `broadcast.created_at >= subscriber.created_at`.
Mark-all-read moves a per-subscriber watermark; it never bulk-updates rows.

## Non-goals

These are deliberate and not on a roadmap to change:

- No workflows, steps, or conditional routing.
- No email, SMS, or push transports. In-app inbox only. (The preference
  `channel` column exists so a push transport needs no migration, but `in_app`
  is the only value today.)
- No server-side templating. Your `payload` is stored and returned verbatim.
- No digests or batching.
