Chimely has two request planes with two auth schemes:

| Plane | Caller | Auth |
|---|---|---|
| **Management** | your backend | `Authorization: Bearer <api-key>` |
| **Subscriber** | `<Inbox />` (one end user) | HMAC subscriber hash |

The admin dashboard is a third plane with its own email/password login and a
session cookie; it is not used by application code.

## Management API keys

Keys are **per environment**. The environment is implied by the key, so
requests carry no environment id. Create and revoke keys in the admin
dashboard (developer or admin role); the plaintext is shown once at creation.
The server stores only its hash plus a short display prefix.

```bash
curl -X POST https://chimely.example.com/v1/notifications \
  -H 'Authorization: Bearer <api-key>' \
  -H 'Content-Type: application/json' \
  -d '{ "subscriber_id": "usr_123", "category": "billing.invoice", "payload": { "title": "Invoice ready" } }'
```

Live keys are prefixed `chml_live_`. Keep them server-side.

<Callout type="warn" title="No CHIMELY_API_KEY variable">
The server reads no `CHIMELY_API_KEY`. Keys are minted in the admin dashboard
and stored in **your** backend's secrets. The only key-related env var is
`CHIMELY_DEV_API_KEY`, which seeds a key into the dev quickstart environment
(it does nothing unless `CHIMELY_DEV_ENVIRONMENT` is also set).
</Callout>

## Subscriber HMAC hash

The widget acts on behalf of one end user. To prove the browser is allowed to
read that subscriber's inbox, your backend computes a hash and passes it to
`<Inbox />`:

```
subscriberHash = hex(HMAC-SHA256(environment_secret, environment_id + "\0" + subscriber_id))
```

The `environment_secret` is the environment's `subscriber_hmac_secret`,
readable in the admin dashboard (developer or admin role). The
`environment_id` is the environment's id (`env_...`), shown on the same
dashboard page; it is joined to the subscriber id with a NUL byte and binds
the hash to that environment, so a hash never transfers between environments
even if two environments were ever given the same secret. Compute the hash in
your backend, per request:

```ts
import { createHmac } from 'node:crypto';

const subscriberHash = createHmac('sha256', process.env.CHIMELY_HMAC_SECRET!)
  .update(`${process.env.CHIMELY_ENVIRONMENT_ID}\0${subscriberId}`)
  .digest('hex');
// pass subscriberHash to <Inbox subscriberHash={...} />
```

<Callout type="warn" title="Legacy formula: accepted for now, removal announced ahead">
Hashes computed with the previous formula,
`hex(HMAC-SHA256(environment_secret, subscriber_id))`, are still accepted as
a fallback. That fallback will be dropped in an announced minor release, and
the cross-environment binding is only effective once it is gone. Switch your
backend to the environment-bound formula now.
</Callout>

<Callout type="error" title="Never sign in the browser">
The secret signs every subscriber's identity for an environment. Computing the
hash client-side leaks it and lets anyone read any inbox. Sign on the server
and send only the per-user hash to the frontend.
</Callout>

The hash travels in request headers (or query parameters for the SSE stream,
where `EventSource` cannot set headers):

| Header | Query fallback | Value |
|---|---|---|
| `X-Chimely-Environment` | `environment` | environment slug |
| `X-Chimely-Subscriber` | `subscriber_id` | your subscriber id |
| `X-Chimely-Subscriber-Hash` | `subscriber_hash` | the HMAC hash |

The `@chimely/client` SDK sets these for you from the `<Inbox />` props.

### Required in production, optional in development

Each environment has a `require_subscriber_hash` flag, **default `true`**.
When true, a request without a valid hash is rejected `401`. The dev-quickstart
bootstrap (`CHIMELY_DEV_ENVIRONMENT`) creates an environment with it `false`,
so the quickstart works without a backend. Turn it on (the default) for any
environment real users touch.

## Rotating the HMAC secret

Rotation is two-slot, zero-downtime. The old and new secrets are both accepted
during the overlap:

1. **Rotate** in the admin dashboard (admin role only; scripting it means `POST /admin/api/environments/{env_id}/hmac/rotate` with an admin session cookie and the `X-Chimely-Admin` CSRF header). A new secret is generated; the previous one stays valid.
2. **Update your backend** to sign with the new secret.
3. **Complete** the rotation (`…/hmac/rotate/complete`). The previous secret is dropped and only the new one is accepted.

Run step 3 once every backend has switched. Hashes signed with the old secret
fail after completion.

## JWT mode

Not available today. HMAC is the only subscriber-auth scheme in 0.x. A JWT
mode is a possible v2 addition, not something that exists now.
